# jesselivingston.com — Site Spec

## Spec Rule
Update this file **before** any code change. Spec first, then code.

## Overview
Static **multi-page** marketing/portfolio site for jesselivingston.com. No build step. One HTML file per page (root-relative links), shared CSS in `/styles.css`, images + the gated PDF in `/assets/`. Deployed via GitHub Pages (push to `main`); DNS on Cloudflare; `CNAME` in repo root.

**Redesigned 2026** to the cream/forest "trust" system (from a Claude design handoff), and split into one page per nav item. This **replaces** the previous dark-purple single-file tabbed site and the embedded Aria prototype, both removed.

## Architecture
- **Pages:** `index.html` (Home: hero + Research-as-Performance + What I've built) · `experience.html` · `portfolio.html` · `services.html` · `contact.html`. Each page is standalone; header (nav) and footer are duplicated (no build/includes); the current page's nav link carries `class="nl active"`.
- **Shared CSS:** `/styles.css` (base + hover/active + the 880px responsive breakpoint), linked from every page with a `?v=` cache-busting query (bump it when CSS changes). The body is a flex column (`footer{margin-top:auto}`) so the footer pins to the bottom on short pages. `header > div` is widened to `1220px` so the nav column aligns with the section content column. Section layout stays as inline styles (each section lives on one page, so nothing is duplicated).
- **Per-page JS:** the password gate is inline on `portfolio.html` only; the Web3Forms handler is inline on `contact.html` only.
- Desktop-first; one responsive breakpoint at **880px** (stacks grids, hero, timeline; hides nav text links). No framework/routing; nav links are root-relative page links (`/about.html` …), wordmark → `/`.

## Portfolio gate (case studies)
- The four case cards and the "Unlock" link are password-gated (soft, client-side). Correct password opens the case-study PDF in a new tab and remembers via `localStorage['jl_pf_ok']='1'`.
- PDF: `/assets/jl-portfolio-8fa31c7e.pdf` (unguessable filename). Password: `livingston2026!` (a `const` in the inline script; configurable).
- **SECURITY (important):** this is a **soft deterrent only**. The PDF is a static file in a public repo, reachable at its URL by anyone who has it; the gate hides the link, not the file. For real protection, host the PDF behind server-side auth / a signed link, or keep it out of the public repo.

## Prototypes
**Saga** is shown as a link to its **hosted live sandbox**; **Aria** and **Corpus** are **self-contained static builds** under `/prototypes/`, committed as built artifacts (no build step on the site; each is rebuilt in its source repo and the output copied here). The two static demos are **backend-free click-throughs**: every AI/API call is intercepted client-side and answered with scripted responses composed from bundled sample data; **no API keys, no network calls beyond Google Fonts**. Each shows a small dismissible "Prototype" badge. Data is fictional/illustrative. Both static demos are **Saga-light themed** (warm canvas, forest accent, Plus Jakarta Sans) to match the site.

**Prototype password gate (soft, client-side).** Aria and Corpus each carry a soft password gate baked into their source `index.html` (survives rebuilds), using the **same password and session key as the portfolio gate** — password `livingston2026!` (case-insensitive), unlock stored in `sessionStorage['jl_pf_ok']`. So unlocking the portfolio also unlocks the prototypes for that session, and vice-versa; hitting a `/prototypes/<slug>/` URL directly prompts for the password. The gate hides the app root (`html.jl-gated #root{display:none}`) behind a cream/forest overlay until the password is entered. Same softness as the portfolio gate (client-side, sessionStorage). Saga is not gated here — its hosted sandbox has its own `demo@` login.

- **Each prototype opens on a landing/intro** (2026). Aria and Corpus each show an in-app splash (product pitch + "Enter" CTA) before the app; Saga's intro is the static `/prototypes/saga/index.html` page below. All three are reached from the same `/prototypes/<slug>/` URL and sit behind the shared password gate.
- **`/prototypes/saga/`** — a **static intro page** (hand-written HTML, not a bundled app) explaining Saga (AI-moderated interviews, rubric scoring, cohort synthesis), showing the shared demo login, and linking out to the hosted sandbox. Same soft gate (`jl_pf_ok`). The Saga cards (home + portfolio) point here, not to onrender directly.
- **Saga (hosted, not bundled)** — the real product's demo sandbox at **`https://saga-dashboard.onrender.com/#/dashboard`** (source: `Saga-Insights/saga-monorepo`, `packages/saga-dashboard`; the "Productivity App Inc." sandbox tenant, `is_sandbox=true`). Access is a **shared demo login**, shown on the site: **`demo@sagainsights.ai` / `saga-demo-2026`**. The tenant is read-only (writes, AI spend, and outbound sends are blocked server-side), so publishing the login is low-risk. **Cannot be iframed** — the app sends `X-Frame-Options: DENY` and CSP `frame-ancestors 'none'`, so it must be a **link-out in a new tab**, not an embed. Hosted on Render's **free tier**, so it **cold-starts (~30s)** after idle; the card notes this. No Saga build lives in this repo.
- **`/prototypes/aria/`** — Aria / Ampli research **communication platform** (source: `JesseLivingston1/ampli`). Vite + React, `base:'./'`; a `window.fetch` shim answers all Claude-API calls with scripted JSON (incl. an SSE stream for the Distill typing effect); localStorage seeded on first visit with the fictional "Pacecar" EV cohort (3 studies, 8 stakeholders, 3 groups); the full 7-step pipeline runs scripted end to end. The **Library view was removed** (2026) so the app is purely the communications platform — nav is **Impact (landing) / Briefs / Drafts / Stakeholders**; the pipeline is reached via a **"View full study →"** action on any Impact row (there is no "+ New Project" button and no Library).
- **`/prototypes/corpus/`** — SubNote Corpus research repository (source: `JesseLivingston1/Corpus`). Vite + React, `base:'./'`; 124 bundled studies; the three AI features (library search, use-case evaluator, per-study Q&A) are answered by a deterministic local mock composing real corpus content. **Reskinned (2026) from its original dark indigo theme to the Saga-light editorial language** (canvas `#f8f7f5`, forest `#123a33`, Plus Jakarta Sans) across all 12 components; WCAG AA verified in-browser. The hardcoded Gemini key the source repo shipped was removed (and should be revoked).
- **`.nojekyll` in repo root** — retained as a safe default so GitHub Pages' Jekyll pass never drops underscore-prefixed asset dirs. (It was originally required for the removed Saga `_next/` export; Aria/Corpus are Vite builds that don't strictly need it, but it's harmless to keep.)
- To update a static prototype: change its source repo, rebuild (`npm run build`), and replace the corresponding `/prototypes/<slug>/` directory wholesale. To update Saga: it's the live hosted app — nothing to rebuild here; just keep the demo login and URL current.
